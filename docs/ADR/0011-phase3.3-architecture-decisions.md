## 决策

### 1. 字号缩放边界：TextSpan 一致性留 Phase 3.4

**决策**：Phase 3.3 字号缩放采用 `MediaQuery.textScaler`（Flutter 内置）方案,**仅保证 `Text` Widget 生效**。Inline `TextSpan` 的硬编码 fontSize 在 Phase 3.3 不做缩放,统一方案留 Phase 3.4 Typography Refactor。

**用户可见影响（需记录,避免 Issue 误报）**：
- Phase 3.3 实施后,用户缩放字号时可能出现：
  - `ParagraphBlock` 正文 → **正常缩放**
  - `InlineBold` / `InlineCode` / `InlineFormula` → **不缩放**
- 这不是 Bug,是 Phase 3.3 已知边界。Exit Gate 不要求 TextSpan 一致性。
- Phase 3.4 Typography Refactor 会统一处理 TextSpan 缩放（考虑方案 A `InheritedWidget` 或方案 C `EditorTokens` 实例化）

**替代方案**：
- 方案 A（`InheritedWidget`）：精确控制每个 token,但 TextSpan 内联渲染需改用 `DefaultTextStyle`,改造量大,Phase 3.3 ROI 不值
- 方案 C（`EditorTokens` 实例化）：破坏 `EditorTokens.paragraphFontSize` 等全项目 `static const` 调用,等于整个渲染层重构,完全不值

### 2. PairInsertCommand：Command Layer 强制路径

**决策**：自动配对（§3.3.6）必须经过 `PairInsertCommand → Coordinator.handle() → Controller 更新` 路径。禁止 `onChanged → 直接修改 controller.text`（违反 Hard Rule 2.3）。

**`PairInsertCommand` 定义**：新增 `EditorCommand` 子类,位于 `lib/presentation/commands/editor_command.dart`（与现有 8 个 Command 同文件,符合 ADR-0008 sealed class 约束）。

**时序说明（R4 评审补充）**：

`onChanged` 在 controller 已被修改后才触发。因此 `PairInsertCommand.apply()` 流程：

```
1. 用户输入 '('
2. controller.text 已变成 "...("
3. onChanged(text) 触发
4. 检测到 '(' 在末尾且需要配对
5. 构造 PairInsertCommand(
     blockId,
     insertOffset: 光标位置,
     pairChar: ')',
     mode: PairInsertMode.appendAfterCursor,
   )
6. Coordinator.handle(command)
7. CommandHandler 生成 UpdateBlockSourceCommand 包含完整 "...()"
8. controller.text 更新为 "...()"
9. 光标移到 '()' 之间
```

**关键约束**：PairInsertCommand 不修改原始用户输入,只追加配对符右半部分。这避免了「先 undo 再 redo」的复杂时序。

**替代方案**：
- `TextInputFormatter` 子类：跨平台一致,但无法感知 Block 类型（CodeBlock 无法禁用配对）
- `FocusNode.onKeyEvent`：移动端 IME（中文/日文/韩文）根本拿不到 KeyEvent,直接淘汰

### 3. Markdown 工具栏 A+B 混合（位置 + 内部布局）

**决策**：Markdown 工具栏采用**位置 A（底部固定栏）+ 内部布局 B（横向滚动）**混合方案,而非单独选择 A 或 B。

**视觉示意**：
```
┌──────────────────────────────────────────┐
│ [B][I][H1][H2][Code][Link][Quote][List] │  ← 横向滚动
│ <------------------------------------->  │
└──────────────────────────────────────────┘
```

**理由**：
- 未来已规划 Table / Mermaid / Image / Divider / Math 等按钮（§3.3.10 模板菜单 + Phase 3.4+ 扩展）,按钮必然超过一屏
- 现在做固定不滚动,后面还得重构
- 直接采用「底部固定位置 + 内部横向滚动」避免未来返工
- Obsidian Mobile / Joplin Mobile 均采用此方案

**参考**：
- 位置选 A：与手机输入法位置接近,操作路径短
- 内部布局选 B：可容纳更多按钮（未来加表格 / 图片 / 分隔线）

### 4. Dirty Tracking 归属 Document State

**决策**：`isDirty` 归入 `CoordinatorState.isDirty` 字段（Document State）,而非独立 Riverpod `UIStateProvider`（UI State）。

**理由**：Dirty 本质属于**文档状态**,不是 UI 状态。以下场景都依赖它：
- AppBar 显示 `•`
- 保存按钮可用
- 关闭时弹窗
- 自动保存判断

归入 Document State（`CoordinatorState`）符合 ADR-0009 §2 BlockViewState / CoordinatorState 的职责边界。

### 5. Toolbar 状态来源：只读 CoordinatorState

**决策**：Markdown Toolbar 只读 `CoordinatorState`,禁止直接访问 `TextEditingController`。

**状态访问路径**：
```
Toolbar
  ↓
Coordinator
  ↓
FocusedBlockState（含 blockId / blockType / selection）
```

**禁止反模式**：
```dart
Toolbar
  ↓
直接访问 TextEditingController  // ❌ 违反依赖方向 chrome → editor
```

**理由**：否则会出现三份选区状态不同步：
- Toolbar 维护一份选区
- Block 维护一份选区
- Coordinator 再维护一份选区

编辑器项目特别容易死在这里。Phase 3.3 必须明确：Toolbar 只读 CoordinatorState,选区 / 聚焦状态由 Coordinator 统一管理。

**新 Command 子类（Phase 3.3 新增）**：

| Command 类 | 用途 | 文件 |
|-----------|------|------|
| `PairInsertCommand` | §3.3.6 自动配对 | `lib/presentation/commands/editor_command.dart` |
| `WrapSelectionCommand` | §3.3.7 选区包裹模式 | `lib/presentation/commands/editor_command.dart` |
| `InsertTemplateCommand` | §3.3.10 模板插入菜单 | `lib/presentation/commands/editor_command.dart` |
| `InsertNewLineWithPrefixCommand` | §3.3.8 自动续列表 | `lib/presentation/commands/editor_command.dart` |
| `InsertTextCommand` | §3.3.7 Markdown 工具栏按钮 | `lib/presentation/commands/editor_command.dart` |

所有新 Command 子类位于同一文件（符合 ADR-0008 sealed class 约束）,并在 `commands.dart` re-export 桥接文件中 show。

## 动机

### 为什么需要本 ADR

ADR-0009 是 Phase 2.9 Prototype 阶段的产物（v1.1 Proposed）,记录的是「UI 架构骨架」决策（BlockViewState / Command Layer / BlockRenderer / 接口冻结）。Phase 3.3 实施时新增的 5 条细化决策（TextSpan 边界 / PairInsertCommand / Toolbar A+B / Dirty 归属 / Toolbar 状态来源）需要独立 ADR,因为：

1. **ADR-0009 状态为 Proposed,尚未 Accepted**：直接在其上叠加新决策会让 v1.1 演进为 v1.2 + v1.3...,决策记录混乱
2. **Phase 3.3 决策具有阶段性**：TextSpan 边界明确声明 Phase 3.4 会移除,PairInsertCommand 是新 Command 类型,与 Phase 2.9 原决策无耦合
3. **Phase 3.3 PR 实施需要明确依据**：实施时需引用 ADR 作为「为什么这样做」的证据,而非凭 Task Contract 推断

### 为什么不更新 ADR-0009

- ADR-0009 仍在 Proposed,Human Owner 未签字 Accepted。AI 在 Proposed 的 ADR 上叠加新决策违反 AGENTS.md §6.4「架构决策类文件 commit 需 Human Owner 授权」
- 新建 ADR-0011 草案让 Human Owner 一次性审批「ADR-0009 是否升级 + ADR-0011 是否新建」两件事

## 后果

### 正面后果

1. **Phase 3.3 PR 实施有明确架构依据**：5 条决策 + 5 个新 Command 子类位置已定义
2. **避免后续踩坑**：TextSpan 边界明确,避免 Phase 3.3 实施时为 InlineBold 不缩放修 Bug
3. **Toolbar 状态来源单一**：避免三份选区不同步问题
4. **Dirty 归属正确**：归入 Document State 而非 UI State,符合职责边界

### 负面后果

1. **ADR-0009 v1.1 与 ADR-0011 并存**：Phase 3.3 实施时需同时引用两份 ADR,可能产生「哪个为准」的疑问
   - 缓解：本 ADR 明确声明「ADR-0009 v1.1 + ADR-0011 共同构成 Phase 3.3 架构依据」
2. **TextSpan 缩放不一致的临时状态**：Phase 3.3 实施后用户会观察到 InlineBold 不缩放现象
   - 缓解：本 ADR §1 明确声明这是已知边界,Issue 误报时引用本 ADR 关闭

### 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| TextSpan 不缩放被用户误报为 Bug | 中 | 本 ADR §1 明确声明 + ROADMAP Phase 3.4 Typography Refactor 接管 |
| PairInsertCommand 时序在 IME 复杂场景失效 | 中 | onChanged 是对 IME 最友好的方案,Phase 3.3 测试覆盖中文输入法组合态 |
| Toolbar A+B 混合在窄屏设备按钮溢出 | 低 | 横向滚动天然支持,用户可滑动找按钮 |
| Dirty 归入 CoordinatorState 后字段过多 | 低 | 相比独立 Provider 引入新状态源,字段增多代价更小 |

## 替代方案

### 替代方案 A：更新 ADR-0009 为 v1.2

- 优点：ADR-0009 成为 UI 架构唯一 ADR,避免双 ADR 并存
- 缺点：ADR-0009 仍 Proposed,Human Owner 未审批。在 Proposed 上叠加新决策违反 AGENTS.md §6.4
- 否决：保留 ADR-0009 v1.1 不变,新建 ADR-0011

### 替代方案 B：不写 ADR,Task Contract §9 即可

- 优点：减少 ADR 数量
- 缺点：Task Contract 是阶段合同,不是长期架构依据。Phase 3.4+ 实施时不会引用 Phase 3.3 Task Contract
- 否决：Phase 3.3 决策具有长期影响（TextSpan 边界 / Toolbar 状态来源）,需 ADR 级别记录

## 未来扩展边界

### Phase 3.4 Typography Refactor 时允许的变更

1. **TextSpan 缩放统一方案**：本 ADR §1 边界声明可移除,采用 InheritedWidget 或 EditorTokens 实例化
2. **Markdown 语义字符智能配对**：带 AST 上下文判断,扩展 PairInsertCommand 或新增 AstAwarePairInsertCommand

### Phase 3.4+ 不允许的变更（需走 ADR 流程）

1. **Toolbar 直接访问 TextEditingController**：本 ADR §5 明确禁止,需新 ADR 授权
2. **绕过 Command Layer 修改 controller.text**：本 ADR §2 明确禁止,违反 Hard Rule 2.3
